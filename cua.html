Common User Access — Unified CUA Specification v1.0

/sys/unified/cua GetLook Edition 1.0 · 2026 · ADR-001–003 · draft

Principles

This is the Unified CUA — Common User Access specification for applications built against the Unified Design System. It is not IBM's 1987 CUA. It inherits from three sources: Plan 9/Acme's interaction model, CDE's keyboard convention, and POSIX accessibility requirements. Where these conflict, the order of precedence is: accessibility > Acme > CDE.

Principle 1 — Text is the interface
Every action available by mouse must be expressible as visible text in the tag bar. If you can't type it, it shouldn't exist only as a gesture. Commands are not hidden in menus — they live in the tag bar where they can be seen, clicked, or keybound.
Principle 2 — User initiates
The system never moves focus without explicit user action. After an HTMX swap, after a validation response, after any server-side event: focus stays where it was. The user decides what happens next. This is the Acme model and also WCAG 2.4.3.
Principle 3 — No mode confusion
There are no hidden modes. What is clickable looks clickable. What is editable signals its editability (pip + underline). What is disabled shows as disabled. The system state is always legible from the screen without memory or inference.
Principle 4 — No animations
State changes are instantaneous. No transitions, no fades, no slides. The heritage is terminal — terminals redraw. The only exception is prefers-reduced-motion-guarded motion that serves a genuine spatial orientation purpose (e.g., a modal appearing). Even then, default off.

Decision 1 — Mouse model: Acme-inspired B1/B2 ADR-001 · decided · 2026-Q2
Button 1 (left click): Select text, position cursor, activate focused element.
Button 2 (right click): Open context menu at pointer position.
Middle click: Execute tag-bar command under pointer if present; paste primary selection in text fields. Never required — always optional.

Full Acme B2-execute and B3-search are not implemented. Tag-bar commands replace B2-execute. Look replaces B3-search. Chording is not defined.
Decision 2 — Keyboard: CDE-style with Acme mnemonics ADR-002 · decided · 2026-Q2
Standard POSIX traversal (Tab/Shift+Tab, Enter, Escape, Space) as the baseline.
Alt+key mnemonics derived from tag-bar command names, not from IBM menu structure.
No modal layers. No vi-style normal/insert split. Keyboard and mouse are always equivalent — neither is primary.
Decision 3 — Focus model: Acme (user-initiated only) ADR-003 · decided · 2026-Q2
Focus is never moved programmatically except by explicit user action or initial page load.
HTMX swaps never auto-focus the swapped content.
Modals (if used) set focus to the first interactive element on open, and return focus to the trigger on close. This is the only server-permitted focus movement.

Drag and drop — permitted extension

Drag and drop is not specified by the Unified CUA and is not a force multiplier across the system’s use cases: Plan 9 has no DnD concept (everything is addressable by path via the plumber), agents do not drag DOM elements, and static documents have no state to reorder. HTMX does not natively support hx-trigger="drop", requiring JS beyond the system baseline.

Implementers may add DnD to specific surfaces (a sortable table, a file upload target) where it provides genuine value. Two requirements apply: (1) a command-based equivalent must always coexist — drag-to-reorder = Move + target path; drag-to-upload = Put with file as body; (2) visual feedback during drag must use border change and background tint only — no animation, consistent with the no-motion principle.

Click targets and hit areas

Every interactive element must have a minimum hit area of 24×24px. Tag-bar commands are inline text — their hit area extends to the surrounding whitespace (4px each side, 2px top/bottom). The worm strip minimum interactive width is 8px even if rendered at 5px (CSS padding or transparent border on the parent achieves this).

✓ Do
Make tag-bar command tokens visually distinct from path text. Color: fg-faint. Hover: bg-sel fill. The visual change is the affordance.
✗ Don't
Use hover-only tooltips as the sole indication that something is clickable. Tooltips are supplementary. The visual state of the element must communicate clickability.
✓ Do
Implement worm strip click-to-scroll. One event listener on the strip element, calculate clickY / stripHeight, multiply by scrollHeight, set scrollTop.
✗ Don't
Implement hover-to-show controls. If an action is available, its trigger must be visible at rest. Hover-reveal violates Principle 3 (no mode confusion).

Context menu

The context menu (right-click / B2) is the bridge between Acme's tag-bar model and platform convention. It must contain only actions that are also available elsewhere — in the tag bar, by keyboard, or by direct manipulation. The context menu is never the sole access point for an action.

Tag-bar grammar

The tag bar is the primary command surface. It contains the window's identity (path or title), universal commands, and application-specific commands. Its grammar is fixed — parseable left-to-right.

Grammar definition
tag-bar := worm-strip path SP* [cmd SP*]* [pipe-cmd SP*]* [flex] [meta] path := identifier for this window's content — filesystem path, URL, or name cmd := bare word, Title-cased, activates a registered command on this window pipe-cmd := '|' bare-word, pipes current selection through a shell filter flex := space that expands to fill remaining width (CSS flex:1) meta := right-aligned status text — modification state, last-saved, etc. Examples: /etc/liquidsoap/radio.liq Put Undo |fmt unsaved /net/tcp/stats Get Kill /search Look [ghost-input] Clear Zonestat Refresh Export Kill
Tag-bar rendering rules
ElementStyleBehaviour
Worm strip ::before pseudo, 5px wide, var(--worm) fill Decorative in the tag bar itself. Interactive on the window body's left edge. See Worm Strip section.
Path font-weight:700, fg-body Not interactive unless the application defines path-click navigation. If clickable, apply standard link underline on hover. Never make path interactive without a visible affordance.
Command token fg-faint at rest, bg-sel on hover, fg-head on hover Click activates. Keyboard: F10 focuses tag bar, Tab moves between tokens, Enter activates. Right-click opens context menu with Look and Get/Put.
Pipe command fg-faint, pipe character in fg-faint Pipes current window selection through the named filter. If no selection, applies to entire window content. Server-side: receives text via POST body, returns replacement text.
Ghost input in tag bar inp-ghost-wrap + inp-ghost Placed after command tokens. Receives focus when user types after focusing the tag bar. Triggers hx-get on keyup with debounce. See Ghost Input section in design system.
Meta / status fg-faint, right-aligned via margin-left:auto Read-only. Communicates modification state, last-updated time, record count. Use a badge for state (modified, saving, error). Use plain text for counts and timestamps.

Focus model

Focus is managed by the user, not the application. The system never moves focus programmatically except in two permitted cases: initial page load, and modal open/close. This is both the Acme model and WCAG 2.4.3 compliance.

Permitted focus movements
EventFocus behaviourRationale
Initial page loadFocus may be set to the primary interactive element (first input in a form, or the main content area's first tag-bar command).Standard browser behaviour. Users expect it.
Modal opensFocus moves to the modal's first focusable element. Focus is trapped within the modal until it closes.WCAG 2.1 SC 1.3.2, 2.4.3. Screen readers and keyboard users cannot access a modal without this.
Modal closesFocus returns to the element that triggered the modal.WCAG 2.4.3. Users must not lose their place in the document.
HTMX swapFocus does not move. The swapped content is rendered; the user's current focus element is unaffected.Acme model. Unexpected focus movement is disorienting and is a WCAG 2.4.3 failure.
Validation responseFocus does not move. Error messages appear adjacent to the field (OOB swap or inline server response). The field retains focus.Acme model. The user is still editing; don't interrupt.
Server errorFocus does not move. Error content swaps into the target. An aria-live region announces the error to screen readers.The user did not ask for a focus change.
Focus order

Tab order follows DOM source order. Do not use tabindex values greater than 0. The only valid tabindex values are 0 (in flow) and -1 (reachable by JS only, e.g. modal trap). Layout that looks logical visually but has illogical DOM order must have its DOM order corrected — not patched with tabindex.

Selection and tag-bar commands

When a tag-bar command is invoked and a text selection exists in the window body or any focused input: the command operates on the selection. When no selection exists: the command operates on the full content of the current window body, or prompts if the command requires explicit input (e.g. Look without a search term).

Aria-live regions

One aria-live="polite" region per page for non-urgent status updates (save confirmation, refresh complete). One aria-live="assertive" region for errors. Both are visually hidden but present in DOM. Server-sent HTMX fragments update these regions as OOB swaps alongside primary content updates.

HTMX interaction

HTMX implements the tag-bar command model over HTTP. The mapping is direct: tag-bar verbs become HTTP verbs, acme-win bodies are swap targets, OOB swaps handle meta and aria-live regions. No JS beyond HTMX and the minimal worm/confirmation helpers.

Agent / util tier

Agents and utilities are second-class consumers of the design system. They receive semantic HTML with correct token-class attributes. They do not need interactive affordances, focus management, or HTMX. What they need: readable structure, correct heading hierarchy, meaningful class names they can parse, and no JavaScript dependencies for content access.

Empty state pattern

An empty state occurs when an acme-win body has no content to display — a search with no results, a list with no items, a resource that has not yet been populated. Empty states must communicate three things: what is empty, why it is empty (or what it would contain), and what the user can do next.

ElementRequiredSpecification
Path in tag-barRequiredThe tag-bar path identifies what is empty. /search being empty is different from /zones being empty.
Explanation lineRequiredOne sentence, monospace, fg-muted. What the space would contain. “No zones configured.” not “Nothing here.”
Suggested actionRequired if action existsA tag-bar command or inline link to the relevant resource. “Run Get to refresh, or create a zone at /zones/new.”
IllustrationForbiddenNo decorative illustration or icon. Heritage: terminals show text, not images.
data-fragment-typeRequired for HTMXdata-fragment-type="window-body" on the empty state container.

Conformance criteria

An implementation conforms to the Unified CUA v1.0 if and only if it satisfies all required criteria below. Recommended criteria improve quality but are not required for conformance. This checklist is the answer to “does this implementation conform?”
