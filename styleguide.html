Tonal palette
All backgrounds and text use Solarized base tones. Body text contrast: 5.61:1 on dark backgrounds, 4.99:1 on light. Both exceed WCAG AA.
Semantic color
Accent colors (primary, success, warning, danger, special, info) are from the Okabe-Ito palette, verified readable under deuteranopia, protanopia, and tritanopia.
Structure
Utility-first composition. Small, orthogonal classes combined at the call site. No component wrappers. No semantic class names for one-off layouts. The terminal heritage: tools, not frameworks.
Application chrome
The CDE titlebar idiom — teal header, logo block, Motif-style controls — is available for application frames. Most views use only a tag bar. Chrome should not compete with content.
utility-first no animations desktop + tablet htmx / static html 2 fonts WCAG 2.1 AA
Design heritage & rationale

The system draws on four workstation traditions: Solarized (Ethan Schoonover, 2011) for its CIELAB-calibrated tonal palette; Okabe-Ito (2002) for its CVD-safe accent colors; Plan 9/Acme (Bell Labs, 1992) for its utility-first structure, tag-bar grammar, and worm strip; and Solaris/CDE (Sun Microsystems, 1993) for its titlebar chrome and Motif depth cues. These were chosen because rigorous perceptual thinking in each tradition independently arrives at solutions compatible with modern accessibility requirements.

Spacing & radius

4px base unit. Eight steps. Radius is a split register: 0px for chrome (Plan 9), 2–6px for content (usability).

Acme patterns — tag bar & body

An acme-win (named for Rob Pike's Acme text editor, Plan 9, 1992) is the fundamental content window: a tag bar for identity and commands, a body for content, and a left-edge worm strip as scroll position indicator. These three parts always compose together. Most pages and views lead with this pattern. The three classes: .acme-win, .tag-bar, .acme-body. Everything else composes from utilities.

Ghost inputs

Ghost inputs are the Plan 9 register — tag-bar-style inline text entry. Two affordance signals prevent misreading as static text: the left-edge pip (3px, --primary, 50% opacity at rest) and the permanent bottom accent line. Hover tints the background; focus makes the pip fully opaque and fills with --bg-sel.

Ghost inputs belong in tag-bar chrome contexts and search affordances — not in data-entry forms.
/* Ghost input anatomy */ <div class="inp-ghost-wrap"> /* ::before = 3px left pip, --primary 50%→100% */ <input class="inp-ghost" placeholder="Look…" hx-trigger="keyup changed delay:300ms" hx-get="/search" hx-target="#results"> </div> /* bottom border: 1.5px solid --primary, always visible */ /* hover: color-mix(--primary 5%) */ /* focus: background: --bg-sel (Acme sel cyan) */

Alerts

Left-border accent, semantic colors. Background tint provides redundant coding alongside the border — color is never the sole signal. color-mix() handles dark-mode opacity automatically.

System information
Unified v2: utility-first, Acme-primary, two fonts, no animations. Dark mode fg-body fixed to base1 (5.61:1).
Deployment complete
Build 1.4.2 promoted. 12 zones updated, zero downtime. fts-quadlet-setup v2.0.3 active.
CVD advisory
Never encode meaning through color alone. Supplement with shape, label, or pattern redundancy.
Token violation
A raw palette constant was used directly. All references must pass through utility classes backed by semantic tokens.

Code

Dark background, monospace, left edge strip. Add .code-block for preformatted blocks. Use fg-syn-* utilities for inline syntax highlighting.

HTMX patterns

The Acme model maps cleanly onto HTMX. Tag-bar commands (Get, Put, Look) become hx-get, hx-post, hx-trigger. The body is the swap target. No JS beyond what HTMX provides.

Live-update window

Usage rules

These resolve ambiguity. They encode decisions already made so they don't have to be made again.

✓ Do
Compose utility classes at the call site. <div class="flex items-center gap-3 p-4 border-lt bg-surface">
✗ Don't
Invent semantic class names for one-off layouts. If it's not in the utility sheet, reach for an inline style before making a new class.
✓ Do
Lead most interfaces with .acme-win / .tag-bar / .acme-body. The CDE bar is for application frames with many top-level actions.
✗ Don't
Use the CDE bar on every page. It signals "application with command menu." Most views are content; those get a tag-bar.
✓ Do
Use .inp-ghost-wrap + .inp-ghost in tag-bar contexts and search affordances. Use .inp in forms and data entry.
✗ Don't
Use ghost inputs without a visible <label>. The pip and bottom line alone are not sufficient affordance for unlabeled entry.
✓ Do
Use r-0 for all chrome elements (titlebars, tag bars, badges, Motif buttons, code blocks). Content cards can use r-2 or r-3.
✗ Don't
Add animation or transition outside of prefers-reduced-motion guarding. Flat state changes only. The heritage is terminal; terminals don't animate.
✓ Do
For dark mode: lift your text reference from --fg-muted to --fg-body or --fg-head if something reads dim. The system is calibrated; trust it.
✗ Don't
Reference raw palette constants (--sol-base1, --oi-blue) in markup. They bypass dark-mode remapping.

Breakpoints

Three breakpoints, desktop/tablet first. These tokens appear in @media (min-width) queries only — they are not part of the token cascade and must not be used as values in component CSS.

--bp-sm768px — tablet portrait floor. Sidebar may collapse.
--bp-md1024px — tablet landscape / small desktop. Full layout available.
--bp-lg1280px — standard desktop. Primary design target.
/* Breakpoint tokens are for @media only — not for var() in component CSS */ /* Desktop-first: start full, restrict down */ @media (max-width: 1023px) { /* below --bp-md */ .page-wrap, .grid-sidebar { grid-template-columns: 1fr; } .page-sidebar { position: static; height: auto; } } @media (max-width: 767px) { /* below --bp-sm — tablet portrait */ .grid-2, .grid-3, .grid-4 { grid-template-columns: 1fr; } .cb { padding: var(--sp-4); } }

Expanded / collapsed state

Disclosure pattern for collapsible sections. No animation — state change is instantaneous. aria-expanded drives the visual indicator and the screen reader announcement. hidden attribute on the body hides content from both visual and accessibility tree.

Readonly state

A field that is populated but not currently editable. Distinct from disabled (which implies permanently unavailable). Readonly: border softens to --border-lt, background shifts to --bg-surface, text is --fg-muted, cursor is default, no focus ring.

Data visualisation colors

Three scales: categorical (Okabe-Ito 7 colors, already defined), sequential (single-hue ramps from Solarized base tones), and diverging (two-hue around a neutral midpoint). All CVD-checked. Use categorical for nominal data, sequential for ordered/continuous, diverging for delta/variance data.

*Yellow (#f0e442) fails contrast on light backgrounds and fails tritanopia. Use only as a supplementary encoding alongside another signal. Never as a primary data color.

Sequential — teal ramp (9 steps, Solarized cyan anchor)

Anchor: Solarized cyan (#2aa198, step 5). Steps 1–4 lighten toward white. Steps 6–9 darken toward base03. CVD-safe: luminance-only progression.

Diverging — vermillion/teal around base2 neutral

Neutral midpoint: Solarized base2 (#eee8d5). Negative pole: OI vermillion (#d55e00). Positive pole: Solarized cyan (#2aa198). Both poles are CVD-safe against each other (distinguishable under all CVD types).
